## 一、JavaScript 执行机制总览

JavaScript 代码执行分为两个阶段：

```txt
1️⃣ 创建阶段（编译阶段）
2️⃣ 执行阶段（运行阶段）
```

1️⃣ 创建阶段会做什么？

在这个阶段，JS 引擎会：

- 创建执行上下文（Execution Context）

- 确定作用域

- 处理变量提升

- 建立作用域链

2️⃣ 执行阶段会做什么？

- 从上到下逐行执行代码

- 进行赋值

- 执行函数调用

## 二、执行上下文（Execution Context）

执行上下文可以理解为：

> 代码运行时的环境

JS 有三种执行上下文：

- 全局执行上下文

- 函数执行上下文

- eval 执行上下文（基本不用）

执行上下文内部包含什么？

一个执行上下文包含：

```txt
1️⃣ 变量环境（Variable Environment）
2️⃣ 作用域链（Scope Chain）
3️⃣ this 绑定
```

## 三、变量提升（Hoisting）

1️⃣ var 的提升

```js
console.log(a)
var a = 1
```

等价于：

```js
var a
console.log(a)
a = 1
```

特点：

- 被提升

- 初始化为 undefined

2️⃣ function 的提升

```js
test()

function test() {
  console.log(1)
}
```

函数声明会整体提升

3️⃣ let / const 的提升

```js
console.log(a)
let a = 1
```

特点：

- 也会提升

- 不会初始化

- 存在暂时性死区（TDZ）

🔥 暂时性死区（TDZ）

- 从作用域开始到变量声明之间的区域。

- 在这个区域访问变量会报错。

## 四、作用域（Scope）

作用域决定变量的可访问范围。

JS 是：

> 词法作用域（Lexical Scope）

也就是说：

- 作用域在写代码的时候就确定

- 不是运行时决定

1️⃣ 全局作用域

在最外层声明的变量。

2️⃣ 函数作用域

```js
function test() {
  var a = 1
}
```

var 受函数控制。

3️⃣ 块级作用域

```js
{
  let a = 1
}
```

let 和 const 受块控制。

## 五、作用域链（Scope Chain）

当访问一个变量时：

> 当前作用域 → 父作用域 → 再往上 → 直到全局

这条查找路径叫：

> 作用域链

示例：

```js
var a = 1

function outer() {
  var b = 2

  function inner() {
    console.log(a)
    console.log(b)
  }

  inner()
}

outer()
```

inner 查找顺序：

```txt
inner 作用域
→ outer 作用域
→ 全局作用域
```

## 六、闭包（Closure）

定义

> 闭包 = 函数 + 该函数能访问的外部变量

更准确的说法：

> 当函数可以访问并持有其词法作用域中的变量时，就形成了闭包。

为什么会产生闭包？

因为：

- 函数执行完 normally 会销毁

- 但如果内部函数被返回

- 外部作用域就不能被销毁

示例

```js
function outer() {
  var a = 1

  function inner() {
    console.log(a)
  }

  return inner
}

var fn = outer()
fn()

```

输出：

```txt
1
```

为什么 a 没有销毁？

因为：

- inner 还在引用a

- JS 垃圾回收机制发现还有引用

- 所以不会回收

闭包的应用场景

1. 数据私有化

2. 模块化

3. 防抖节流

4. Vue 响应式原理

## 七、for + setTimeout 原理

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i)
  }, 0)
}

```

输出：

```txt
3
3
3
```

原因：

- setTimeout 是宏任务

- for 循环先执行完

- i 变成 3

- 回调访问的是同一个 i

let 版本：

```txt
0
1
2
```

原因：

- 每次循环创建新的块级作用域

- 每个 i 独立存在

类似于

```js
{
  let i = 0
  setTimeout(...)
}
{
  let i = 1
  setTimeout(...)
}
{
  let i = 2
  setTimeout(...)
}
```

```txt
执行上下文

 │     ├── 变量提升

 │     └── 作用域链建立

```

```js
var a = 1

test2()

```

## 进阶理解：内存角度解释闭包

```js
function outer() {
  var a = 1
  return function () {
    console.log(a)
  }
}

var fn = outer()
fn()
```

为什么 a 没被销毁？

正常情况：

> 函数执行完 → 执行上下文出栈 → 变量销毁

但这里：

> inner 还在引用 a

所以 JS 引擎会：

> 把 outer 的变量环境保存到堆内存

这就形成闭包。

## 进阶题 1

```js
function test() {
  var a = 1
  return function() {
    a++
    console.log(a)
  }
}

var fn1 = test()
fn1()
fn1()
fn1()
```

输出什么？

为什么？

### 🔥 正确底层解释（面试标准版）

当执行：

> var fn1 = test()

执行流程：

① 进入 test 执行上下文

创建变量环境：

> a = undefined

执行阶段：

> a = 1
> return inner

此时：

- inner 函数引用了变量 a

- 所以 test 的变量环境不会被销毁

- 被保存在堆内存中

② 每次执行 fn1()

都会访问同一个 a

> a++

因此：

> 2 → 3 → 4

⚠ 你有一个表达错误

你说：

保存到堆栈

❌ JS 只有：

栈（执行上下文）

堆（对象、闭包环境）

闭包的变量环境存在 堆内存，不是“堆栈”。

面试中这样说：

> 函数执行完本应出栈销毁，但因为内部函数仍引用外部变量，JS 会将其变量环境保存在堆内存中，形成闭包。

## 进阶题 2

```js
function test() {
  var a = 1
  return function() {
    console.log(a)
  }
}

var fn1 = test()
var fn2 = test()

fn1()
fn2()
```

输出什么？

fn1 和 fn2 共享同一个 a 吗？

为什么？

### 为什么不共享？

每调用一次：

> test()

都会：

- 创建新的执行上下文

- 创建新的变量环境

- 生成新的 a

所以：

> fn1 → 闭包1 → a(1)
> fn2 → 闭包2 → a(1)

是两个独立的闭包。

🔥 面试高级表达

- 每次调用 test 都会创建新的执行上下文，因此闭包持有的是不同的词法环境，变量互不影响。
