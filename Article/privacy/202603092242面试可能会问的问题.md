## 请你区分一下 var、let、const

- var、let、const 的区别主要体现在作用域、变量提升和赋值特性上。
- var 是函数作用域，存在变量提升，声明前访问得到 undefined；let 和 const 是块级作用域，也会提升，但在声明前处于暂时性死区，不能访问。
- 另外，var 允许重复声明，let 和 const 不允许。
- 赋值方面，let 可以重新赋值，const 不可以，但如果 const 保存的是对象，修改对象属性是可以的，因为它没有改变引用地址。

## 说一下什么是暂时性死区？为什么 let 和 const 会有这个机制？

- 暂时性死区是从作用域开始到变量声明之间的区域,再这个区域访问变量会报错.
- 暂时性死区是指从块级作用域开始，到 let 或 const 声明之前的这段区域。
- 在这段区域内，变量虽然已经在作用域中存在，但不能被访问，访问就会报错。
- 这样设计主要是为了避免像 var 那样因为变量提升导致声明前访问得到 undefined，从而提高代码的安全性和可读性。
- 所以可以理解为：let 和 const 也有提升，但在声明前不会初始化。

## let、const 有变量提升吗？如果有，为什么声明前不能访问？

- let 和 const 也有变量提升，但和 var 不一样。
- var 提升后会初始化为 undefined，所以声明前访问不会报错；let 和 const 提升后不会初始化，在声明前处于暂时性死区，所以不能访问。
- 也就是说，它们不是“不提升”，而是“提升了但不能提前用”。

## 说一下闭包是什么？闭包有什么作用和缺点？

- 闭包就是内部函数引用了外部函数作用域中的变量，导致外部函数执行结束后，这些变量仍然可以被访问。
- 它本质上是函数和其词法作用域的组合。
- 闭包常见作用是实现数据私有化、保存状态、做模块化封装。
- 缺点是如果滥用，可能导致变量不能及时被垃圾回收，增加内存开销。

## 闭包为什么会造成内存泄漏？它一定会造成内存泄漏吗？

- 闭包可能导致内存占用增加，是因为内部函数引用了外部变量，只要这个闭包还被引用，这些变量就是可达的，垃圾回收器就不会回收它们。
- 但闭包本身不等于内存泄漏，它只是可能让某些数据存活时间更长。
- 如果闭包是业务上正常需要的，用完后也及时释放引用，那就没有问题。
- 真正的风险是在不再需要时还长期持有闭包，导致内存不能及时释放。

## 说一下 JavaScript 中的原型和原型链

- 在 JavaScript 里，每个构造函数都有一个 prototype 属性，它指向一个原型对象。
- 通过这个构造函数创建出来的实例对象，内部会有一个指针指向这个原型对象，通常可以理解为
  **proto** 指向构造函数的 prototype。
- 原型对象的作用主要是实现属性和方法的共享。
- 比如某个方法定义在原型上，那么这个构造函数创建出来的所有实例都可以访问这个方法，不需要每个实例都复制一份。
- 原型链指的是对象在查找属性或方法时的查找路径。
- 当访问一个对象的属性时，会先在对象自身上找；如果没找到，就沿着它的原型对象继续找；如果还没找到，就继续往上一层原型找，直到 Object.prototype，再往上就是 null。
- 这种一层层向上查找形成的链式结构，就叫原型链。

## 对象查找一个属性时，具体会按照什么顺序去找？

- 对象查找属性时，首先会先在对象自身上查找。
- 如果对象本身没有这个属性，就会去它的原型对象上找。
- 如果原型对象上也没有，就继续沿着原型链向上查找，也就是查找“原型的原型”。
- 这个过程会一直找到 Object.prototype，如果还没有，再往上就是 null，说明原型链到头了。
- 这时候如果还是没找到，最终返回 undefined。

## prototype 和 **proto** 有什么区别？

- prototype 是构造函数上的属性，指向原型对象；
- **proto** 是实例对象内部指向原型对象的链接。
- 构造函数通过 prototype 提供共享属性和方法，实例对象通过 **proto**
  连接到原型链上，实现属性的向上查找。

## 说一下 this 的指向规则

- this 的指向不是在定义时决定的，而是大多数情况下在函数调用时决定的，主要取决于函数的调用方式。
- 对于普通函数，this 的指向主要有四种规则：
- 第一是默认绑定。如果函数独立调用，在非严格模式下 this 指向全局对象，在严格模式下是 undefined。
- 第二是隐式绑定。如果函数作为对象的方法调用，this 指向调用它的那个对象。
- 第三是显式绑定。可以通过 call、apply、bind 主动指定 this`` 指向。
- 第四是new 绑定。如果函数通过 new 调用，this 会指向新创建的实例对象。
- 另外，箭头函数没有自己的 this，它的 this 由外层作用域决定，一旦确定就不能通过 call、apply、bind 改变。
- 如果同时存在多种绑定，一般优先级可以理解为：new 绑定 > 显式绑定 > 隐式绑定 > 默认绑定。箭头函数则不走这套规则，它直接继承外层 this

## 箭头函数为什么没有自己的 this？它的 this 是怎么确定的？

- 箭头函数没有自己的 this，这是它和普通函数很大的区别。
- 普通函数的 this 一般是在调用时决定的，而箭头函数的 this 不是在调用时绑定的，而是在定义时就确定了。
- 它会继承外层词法作用域中的 this，并且这个指向一旦确定，就不能通过 call、apply、bind 去改变。
- 这样设计的好处是可以避免在回调函数或嵌套函数中出现 this 丢失的问题。
- 比如在对象方法里写一个箭头函数回调，它通常会直接继承外层方法的 this，不需要额外写 var
  that = this。

## JavaScript 的事件循环（Event Loop）是什么？宏任务和微任务有什么区别？

- JavaScript 是单线程的，主线程一次只能做一件事。
- 为了处理异步任务，浏览器或 Node 这样的宿主环境会把一些异步操作交给对应的模块去处理，比如定时器、网络请求、事件监听等。当这些异步任务完成后，对应的回调不会立刻执行，而是进入任务队列，等待主线程空闲后再执行。
- Event
  Loop 的作用，就是不断检查调用栈是否为空，如果为空，就从任务队列中取出任务放到主线程执行，这个过程就叫事件循环。
- 在任务类型上，通常分为宏任务和微任务。常见宏任务有：script、setTimeout、setInterval、I/O、页面渲染等。
- 常见微任务有：Promise.then/catch/finally、MutationObserver，以及 Node 里的部分微任务。
- 它们的执行顺序是：先执行同步代码；同步代码执行完后，先清空当前轮次的微任务队列；再去执行一个宏任务；执行完这个宏任务后，再清空微任务队列；然后进入下一轮循环。
- 所以可以简单理解为：每一轮事件循环里，微任务的优先级高于宏任务。

## == 和 === 有什么区别？为什么面试里一般建议用 ===？

- == 和 === 的主要区别在于是否会进行隐式类型转换。
- == 在比较时，如果两边类型不同，JavaScript 会先尝试做类型转换，再判断它们是否相等。
- 所以像 '1' == 1、true == 1 这种结果都会是 true，这就容易引起歧义。
- 而 === 是严格相等，不会进行类型转换，只有当两边的类型相同且值也相同时，结果才是 true。
- 面试和实际开发中一般更推荐使用 ===，因为它的判断更明确、可读性更好，也能避免隐式类型转换带来的 bug。

## 说一下深拷贝和浅拷贝的区别，常见实现方式有哪些？

- 浅拷贝和深拷贝主要是针对引用类型来说的。
- 浅拷贝指的是：只复制对象的第一层属性。如果某个属性还是对象或数组，那么复制的只是它的引用地址，多个对象可能会共享同一份内部数据，所以修改嵌套对象时会互相影响。
- 深拷贝指的是：递归地把对象的每一层都复制一份，拷贝后得到的是一个完全独立的新对象，修改新对象不会影响原对象。
- 常见的浅拷贝方式有：Object.assign();展开运算符 ...;数组的 slice()、concat()
- 常见的深拷贝方式有：JSON.parse(JSON.stringify(obj)),lodash.cloneDeep,手写递归实现深拷贝
- 不过 JSON.parse(JSON.stringify()) 也有缺点，比如它会丢失 undefined、function、Symbol，并且不能正确处理 Date、RegExp、循环引用等情况。
